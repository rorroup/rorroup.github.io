Projection

WebGL allows for high quality rendering to an html canvas element. It outputs from an internal 4-dimentional normalized coordinate system regarded as clip space ranging from -1 to 1 to the screen by scaling the contents to fit the destination resolution. Points for drawing are passed into WebGL using special programs called shaders. Firstly, the vertex shader defines the position of the desired point by setting a special variable called gl_Position using a 4-component vector in what is called homogeneous coordinates meaning that the first 3 components correspond to the cartesian x, y and z coordinates dictating horizontal position, vertical position and depth respectively, while the fourth component usually called w serves as a factor whose inverse scales the whole vector; so that everything laying within the 3-dimentional origin-centered 2-unit cube will be drawn and everything else will be ignored. Then the color of the point is established by setting another special variable called gl_FragColor in the fragment shader.
The rendering of a 3D scene requires mapping the vertices of the objects in the world to the clip space so that WebGL knows how to draw them to the screen. This is generally achieved using linear algebra through the application of linear transformations consisting in matrix multiplication to manipulate space thus driving the points to their appropiate coordinates. Being an electrical engineer I'm a big fan of linear systems myself.
Square 3x3 matrices allow for a variety of transformations of three dimentional vectors. Scaling for example implies the multiplication of the vector's components by a constant value. Let P = ( P x , P y , P z ) be a vector in cartesian space and k the scaling constant. Then its scaling matrix satisfies the equality ( x 0 x 1 x 2 y 0 y 1 y 2 z 0 z 1 z 2 ) ( P x P y P z ) = ( x 0 P x + x 1 P y + x 2 P z y 0 P x + y 1 P y + y 2 P z z 0 P x + z 1 P y + z 2 P z ) = ( k P x k P y k P z ) = k ( P x P y P z ) giving the solution ( k 0 0 0 k 0 0 0 k ) = k ( 1 0 0 0 1 0 0 0 1 ) which can be written as kI3 using the identity matrix. Rotation may be similarly deduced among other transformations. It can be proved however that translation by a vector T = ( T x , T y , T z ) can not be accomplished in this fashion since the matrix equation ( x 0 x 1 x 2 y 0 y 1 y 2 z 0 z 1 z 2 ) ( P x P y P z ) = ( x 0 P x + x 1 P y + x 2 P z y 0 P x + y 1 P y + y 2 P z z 0 P x + z 1 P y + z 2 P z ) = ( P x + T x P y + T y P z + T z ) = ( P x P y P z ) + ( T x T y T z ) has no solution. This issue is circumvented by conveniently adding the aforementioned 4th component w yielding ( x 0 x 1 x 2 x 3 y 0 y 1 y 2 y 3 z 0 z 1 z 2 z 3 w 0 w 1 w 2 w 3 ) ( P x P y P z 1 ) = ( x 0 P x + x 1 P y + x 2 P z + x 3 y 0 P x + y 1 P y + y 2 P z + y 3 z 0 P x + z 1 P y + z 2 P z + z 3 w 0 P x + w 1 P y + w 2 P z + w 3 ) = ( P x + T x P y + T y P z + T z 1 ) = ( P x P y P z 1 ) + ( T x T y T z T w ) where for simplicity the w component of the 4-dimentional homogeneous equivalent of the P vector P H = ( P x , P y , P z , P w ) as well as the translated vector have been forced to scale by 1 so that the homogeneous coordinates match numerically their 3-dimentional counterparts. The solution to this equation is the 4x4 3D translation matrix ( 1 0 0 T x 0 1 0 T y 0 0 1 T z 0 0 0 1 ) that is clearly independent of the Tw translation component which is usually considered as 1 for consistency reasons as it won't affect the numerical result. This approach establishes the points are to be passed as 4D homogeneous vectors and transformed by 4x4 matrices. Other transformation matrices may be expanded from 3x3 to 4x4 rather trivially like the scaling matrix ( k 0 0 0 0 k 0 0 0 0 k 0 0 0 0 1 ) keeping in mind a 3D vector is numerically equivalent to its homogenooeus expansion scaled by w = 1. This enables composing multiple matrix multiplications to achieve further transformations.
The 2D screen pixel coordinate for rendering a particular point that exists in the world is calculated from its mapping into WebGL 4D clip space obtained from the perspective of an observer looking into the 3D space through a process called projection. Therefore it is only natural to begin with building a sufficiently accurate model to mimic human sight. Biology is not my specialty, nor is optics, but we know we see by percieving the light coming trough the pupil of the eye which bounced off of an object allowing us to see it. Therefore I establish 2 fundamental laws: firstly light travels in a strictly straight line; and secondly, that which can be seen is light that reaches a particular point in space from the front as if it was the pupil which I will call the camera. To define the camera's front and for simplicity and without loss of generality the camera will exist at the origin of my coordinate system with the x axis going to the right and the y axis going upwards so that the center of the view as well as the x and y axis match those of the clip space. Again, being an electrical engineer I'm particularly fond of the right hand rule too, which implies the z axis points backwards meaning the camera is looking straight down the negative z axis.
Having determined where the camera is looking at the field of view angle FoV is defined to adequately asses whether points are visible in front of it. It spreads vertically and symmetrically centered on the camera's axis so that only light inciding on the camera at an angle within its edges can be seen. This vertical angle naturally imposes an opposite vertical visible range in the y direction increasing the further away from the camera controlling the visibility of points in space so as to exclude those directly above or even behind it. Since the screen is a rectangle there obviously must exist a corresponding horizontal visible range related by the screen's aspect ratio AR = Screenheight / Screenwidth . The resulting rectangular boundary defines a rectangular orthogonal symetric pyramid extending indefinitely in the negative z direction with the camera at its tip looking down its center. The ever growing rectangles down the pyramid normal to the camera direction are naturally all similar among them and are also similar to the screen.
Because computers belong to the real world they inevitably have finite capabilities so our camera can only see so far. We set a plane normal to the camera direction at coordinate negative Zfar as the furthest visible distance whose intersection with the field of view's edges delimits the rectangle to serve as the base of the pyramid. Lastly a second plane parallel to the pyramid's base is added closer to the camera at a distance Znear so nothing closer is rendered since everything would collapse on a single point as approaching the camera.
The resulting volume bounded by the vertical field of view, its corresponding horizontal ratio, and the near and far planes has the shape of a frustum and corresponds to the visible volume to be projected onto the clip space for rendering. The best analogy I can think of is as if looking at your screen is the same as looking into the world trough a window. We are standing still at a fixed distance before the window and everything in the world happens beyond it so all we can see is what is directly in front of it at any given time.

The 4x4 projection matrix is built from the geometrycal relationships to accurately map a world coordinate W = ( W x , W y , W z , 1 ) into normalized clip space. Since our model has light traveling in a straight line rays at the field of view's edges form an isosceles triangle whose height Znear bisects the FoV and whose base lies on the near plane with length Hnear = 2 hnear holding the relationship tan ( FoV 2 ) = hnear Znear Because the rectangular orthogonal symmetric visible frustum has its tip in the camera located at the origin implying it is 0 centered in both the x and y directions a ray from the camera to the point in the world W intersects the near plane at a point N = ( N x , N y , N z , 1 ) proportional to it in 3D forming 2 pairs of similar triangles in both the x and y directions. A glaring problem rises though since both the x and y clip components are to be scaled by a constant value divided by the vector's own Wz coordinate in circumstances where it is clearly impossible to accomplish that by matrix multiplication. Here is where homogeneous coordinates H = ( H x H y H z H w ) = 1 C w ( C x C y C z C w ) come in handy. Since in the end the whole vector will get scaled internally its value is conveniently set to Cw = Wz allowing to accomodate the previous equations to the homogeneous result H yielding Hx = 1 Cw Cx = 1 Wz AR tan ( FoV 2 ) Wx Hy = 1 Cw Cy = 1 Wz 1 tan ( FoV 2 ) Wy The projection matrix is therefore constructed in the form ( AR tan ( FoV 2 ) 0 0 0 0 1 tan ( FoV 2 ) 0 0 0 0 α β 0 0 -1 0 ) where the z component has been negated since its value grows negative in front of the camera so as to not flip the x and y coordinates. The depth normalization has also been assumed to not depend on the x and y components so their respective factors have been set to 0.
